# 펫 스트리밍 → WindowServer 부하 진단 (2026-07-11)

맥북 체감 렉의 원인 조사 결과. 메모리·발열·디스크 전부 정상이었고, 범인은 펫 플러그인의 스트리밍 갱신이 유발하는 WindowServer(화면 합성 프로세스) 포화였다.

## 실측 (M-시리즈 MBP 14", 3024×1964 내장 디스플레이 단독)

| 상태                        | WindowServer CPU | SwiftBar CPU | 펫(bun) CPU |
| --------------------------- | ---------------- | ------------ | ----------- |
| 펫 실행 중 (0.5~1s 틱)      | **102~117%**     | ~25%         | ~0%         |
| SwiftBar 완전 종료          | 38~51%           | —            | —           |
| SwiftBar 재시작 + 펫 비활성 | **26~28%**       | 5.7%         | —           |

- 메모리 free 93%, swap 0, 발열 스로틀링 기록 없음 → 렉은 순수 WindowServer 포화.
- 펫 하나가 WindowServer 약 **+80%p** + SwiftBar **+20%p**를 만들고 있었다.

## 원인 분석

1. **프레임 생성 비용은 0이 맞다.** 스프라이트는 `spriteCache`에 PNG base64로 미리 렌더·캐시되고, 매 틱은 문자열만 꺼내 출력한다. bun 프로세스 CPU 0% 실측으로 확인.
2. **비용은 출력 이후 파이프라인에서 발생.** SwiftBar streamable은 diff 개념이 없다. `~~~` 프레임마다:
   - 출력 전체 재파싱 (이미지 base64 디코드 + 드롭다운 12줄 재구성)
   - NSMenu 전체 리빌드 + NSStatusItem 이미지 교체 → 메뉴바 아이템 무효화
   - WindowServer가 메뉴바 재합성 (Retina 3024px 폭 + 블러 머티리얼)
3. **2프레임 교대가 함정.** A→B→A→B로 매 틱 프레임이 항상 바뀌므로 출력을 스킵할 수 있는 틱이 없다. working 상태 500ms 틱 = 초당 2회 풀 리빌드.
4. 갱신 1회당 비용이 비정상적으로 큰 것(초당 1~2회에 +80%p)으로 보아, 병목은 macOS 합성 자체보다 **SwiftBar의 streamable 프레임 처리 경로**(사실상 상태 아이템 전체 재구성)로 추정. 이 비용은 플러그인 쪽에서 제어 불가 → 레버는 emit 빈도뿐.

## 부수 발견

- **`.js.off` rename은 비활성화가 아니다.** SwiftBar는 `.off` 파일도 그대로 실행한다 (좀비 스트림 실측 — rename+kill 후에도 SwiftBar가 재스폰). 비활성화는 반드시 `swiftbar://disableplugin?name=<파일명>` URL 스킴. CLAUDE.md 도메인 지식 항목과 일치.

## 수정 방향 (v1.15.0)

원칙: **애니메이션은 봐줄 사람이 있을 때(활동 중)만, 나머지는 느리게 또는 정지.**

1. 출력 diff 게이트 — 직전 emit과 출력이 동일하면 `console.log` 스킵 (단일 프레임 상태 tired/exhausted는 사실상 정지). 스킵하더라도 상태 파일 폴링은 계속해 반응성 유지.
2. 상태별 틱 간격 재조정 — working/party/이벤트(8s 한정)만 빠르게, idle은 수 초, sleep은 더 느리게.
3. 목표: 평상시 WindowServer 기여 한 자릿수 %p.

수정 후 동일 방법(`ps -o %cpu -p $(pgrep -x WindowServer)` 3샘플)으로 전후 비교할 것.
